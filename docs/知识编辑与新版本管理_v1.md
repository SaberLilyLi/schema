# 需求设计文档：知识编辑与新版本管理（v1）

## 1. 目标

在现有“新建知识 / 查看详情 / 审批发布”基础上，新增“编辑已有知识并生成新版本”能力，形成完整闭环：

- 草稿可反复编辑
- 被驳回后可修改再提审
- 已发布知识可生成新版本进入再审批流程

## 2. 角色与权限

- 访问编辑入口：`kb:edit`
- 提交审批：`kb:submit`（沿用现有）
- 审批通过/驳回：`approval:approve`（沿用现有）
- 发布：`kb:publish`（沿用现有）

## 3. 功能范围（v1）

### 3.1 详情页新增“编辑”入口

- 在知识详情页增加“编辑”按钮
- 按状态控制按钮可见性：
  - `draft` / `rejected`：可直接编辑当前版本
  - `published`：点击编辑后创建“新版本草稿”（`versionNo + 1`）
  - `submitted` / `approved`：默认不允许直接改正文（避免审批中篡改）

### 3.2 编辑页面支持“编辑模式”

- 复用 `ArticleEditorView`，支持两种模式：
  - 新建模式（已有）
  - 编辑模式（新增）
- 编辑模式打开时自动回填：
  - 标题、条线、密级、变更说明、正文
- 保存时分两类：
  - 更新草稿（同版本覆盖）
  - 已发布知识生成新版本草稿（新建 version）

### 3.3 版本列表增强

- 在详情页版本抽屉中标注“当前版本”
- 显示每个版本的流程状态、提交/审批/发布时间（已有字段补齐展示）

## 4. 后端设计（拟新增）

## 4.1 接口

1. `POST /api/articles/:id/editable`
   - 作用：获取可编辑数据与编辑策略（是否新开版本）
   - 权限：`kb:edit`

2. `POST /api/articles/:id/save`
   - 作用：保存编辑内容
   - 权限：`kb:edit`
   - 规则：
     - 当前为 `draft/rejected`：更新当前版本内容
     - 当前为 `published`：创建 `versionNo + 1` 草稿并切换 `currentVersionId`

## 4.2 审计日志

- `kb.edit.read`：进入编辑前读取可编辑数据
- `kb.edit.save`：保存编辑内容
- `kb.version.create`：已发布知识创建新版本草稿

## 5. 前端设计（拟新增）

## 5.1 页面与路由

- 路由新增：`/knowledge/:id/edit`（权限 `kb:edit`）
- `ArticleDetailView` 增加“编辑”按钮并跳转编辑路由
- `ArticleEditorView` 支持 `route.params.id` 编辑模式

## 5.2 交互细节

- 保存成功后：
  - 编辑草稿：回到详情并提示“已保存草稿”
  - 已发布创建新版本：提示“已创建新版本草稿”
- 提交审批按钮逻辑与当前 `workflowState` 联动（沿用现有）

## 6. 验收标准

- 草稿/驳回知识可编辑并保存成功
- 已发布知识可创建新版本草稿并出现在版本列表
- 审批中/已审批状态下禁止正文直接改动（按策略提示）
- 审计日志中能查到编辑读取与保存动作
- 前后端构建通过，核心流程可跑通

## 7. 不在 v1 范围

- 富文本协同编辑
- 版本差异对比（diff 视图）
- 回滚到指定历史版本
