# 需求设计文档：审计中心模块（v1）

## 1. 背景与目标

当前系统已在关键操作（登录、知识检索、提交流程、审批、发布）写入审计日志，但缺少统一的查询与查看入口。  
本模块目标是提供“审计中心”页面，支持按条件检索、分页浏览、查看详情，满足运维排障与合规追踪。

## 2. 适用角色与权限

- 可访问角色：具备 `audit:read` 权限的用户（如 `kb_admin`、`auditor`）
- 菜单展示规则：左侧菜单“审计中心”仅在用户具备 `audit:read` 时可见
- 路由访问规则：`/audit-center` 路由受 `audit:read` 保护

> 后续会同步更新 `docs/ROLE_ACCOUNTS.md` 的菜单权限映射与角色权限清单。

## 3. 功能范围（v1）

### 3.1 审计日志列表

支持字段展示：

- `seq`（序号）
- `occurredAt`（发生时间）
- `action`（动作）
- `actorUsername`（操作人）
- `resourceType`（资源类型）
- `resourceId`（资源标识）
- `success`（是否成功）
- `statusCode`（状态码）
- `requestId`（请求链路ID）

### 3.2 查询条件

- 时间范围（开始/结束）
- 操作人（用户名，模糊）
- 动作（精确或前缀）
- 资源类型（精确）
- 请求ID（精确）
- 成功状态（全部/成功/失败）

### 3.3 日志详情抽屉

点击列表行打开详情，展示：

- 基本信息（时间、操作人、IP、UA、动作、资源）
- 请求信息（`requestId`、`statusCode`、`errorCode`）
- `details` JSON（格式化展示）
- 哈希链字段（`prevHash`、`recordHash`）用于完整性追踪

## 4. 非功能要求

- 列表查询接口响应目标：常规条件下 < 500ms（本地开发环境）
- 默认分页：`page=1`，`pageSize=20`
- 最大分页大小：`100`
- 时间查询采用后端统一时区解析（ISO 字符串）

## 5. 后端设计（拟新增）

## 5.1 路由

- `POST /api/audit/logs`
  - 权限：`audit:read`
  - 功能：分页查询审计日志
- `POST /api/audit/logs/:id`
  - 权限：`audit:read`
  - 功能：获取单条日志详情

## 5.2 查询参数（`POST /api/audit/logs`）

- `page` number
- `pageSize` number
- `actor` string（用户名模糊匹配）
- `action` string
- `resourceType` string
- `requestId` string
- `success` boolean（可选）
- `startAt` string（ISO）
- `endAt` string（ISO）

## 5.3 返回结构（沿用现有 API Envelope）

- `items`: 列表数据
- `page` / `pageSize` / `total`

## 6. 前端设计（拟新增）

## 6.1 页面与路由

- 新页面：`frontend/src/views/audit/AuditCenterView.vue`
- 新路由：`/audit-center`
- 左侧菜单新增：审计中心（权限 `audit:read`）

## 6.2 组件复用

- 页面框架：`commonPages`
- 查询区：`commonSearch`
- 列表区：`commonTable`

## 6.3 交互流程

1. 进入页面默认按时间倒序加载
2. 用户设置查询条件点击“查询”
3. 分页切换保持当前过滤条件
4. 点击行打开详情抽屉

## 7. 验收标准

- 无 `audit:read` 权限用户看不到菜单，访问路由返回无权限页
- 有权限用户可完成列表查询、分页、详情查看
- 查询参数与返回字段与文档一致
- 前端构建通过，后端接口可用

## 8. 不在 v1 范围

- 审计日志导出（CSV/Excel）
- 审计日志归档与自动清理策略
- 双人授权（4-eyes）删除/迁移审计日志
